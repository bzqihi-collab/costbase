🏗️ 三种技术方案对比

需求回顾：跨平台桌面应用 | 离线查询 | 定期联网同步 | 5-10人 | 单机版

⭐ A

Electron + React + SQLite（推荐）

用 Web 技术构建桌面应用，SQLite 存本地数据，Node.js 跑同步任务

👍 优点

  • 真正的桌面应用体验（菜单栏、托盘、快捷键）
  • React 生态成熟，UI 可以做得很好看
  • SQLite 轻量高效，离线查询毫秒级响应
  • Node.js 有丰富的 HTTP 客户端库对接外部 API
  • 可打包成安装包（.dmg / .exe / .AppImage）

👎 缺点

  • 安装包较大（~150MB）
  • 内存占用相对较高（~200MB）
  • 开发周期中等（4-6周MVP）
B

Python 后端 + 本地 Web + 桌面壳

Python 做数据同步和查询引擎，浏览器做 UI，用轻量壳包装成桌面应用

👍 优点

  • Python 数据处理生态最强（pandas、爬虫、API对接）
  • DuckDB 分析型查询比 SQLite 更快（聚合/对比）
  • 开发速度快（2-4周MVP），调试方便
  • 前端可用任何框架，或直接用 Streamlit 省掉前端开发

👎 缺点

  • 需要启动本地服务器，用户感知明显
  • 浏览器访问不如原生桌面自然
  • Streamlit 方案 UI 灵活性有限
  • Python 打包成桌面应用（PyInstaller）体验一般
C

Tauri + Vue/Svelte + Rust 同步引擎

Tauri 做轻量桌面壳，前端用 Vue/Svelte，Rust 处理数据同步

👍 优点

  • 安装包极小（~10MB），内存占用低
  • 性能优秀，启动速度快
  • 安全性好，Rust 内存安全

👎 缺点

  • Rust 学习曲线陡峭，维护成本高
  • 数据对接/爬虫生态远不如 Python/Node
  • 小团队如果没人懂 Rust，不可行
  • 开发周期长（6-10周）

推荐理由：方案 A

Electron + React + SQLite 是桌面应用 + 离线查询场景最成熟的方案。VSCode、Notion、Figma 都用它。

数据同步部分，Node.js 完全能胜任 API 对接和定时任务。5-10 人单机使用，不需要考虑服务端并发，SQLite 在本地查询性能足够（几十万条数据毫秒响应）。
